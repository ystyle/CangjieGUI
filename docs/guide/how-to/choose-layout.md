# 为界面选择布局容器

## 目标

根据内容关系选择纵向、横向、网格、流式、层叠、滚动或分栏容器，并说明哪些区域应伸缩。完成后，窗口放大和缩小时仍保持阅读顺序与主次关系，而不是只在设计时尺寸下勉强对齐。

## 适用场景

在把表单扩展成工具栏、侧栏、指标卡、长内容或主从工作区时使用。若主要问题是大量数据项的性能和身份，先选出大结构，再到[数据列表](data-list.md)决定普通滚动还是虚拟化。

## 准备工作

先保留[设置表单](../tutorials/settings-form.md)的完整程序；本页的片段都说明替换它的哪一段。画出内容区块并标注关系：上下阅读、左右协作、二维等宽、可换行标签、同一区域前后叠放、超出视口、可拖动工作区。再标出固定尺寸、内容尺寸和应占剩余空间的区域。不要先从控件类型出发；同一个 `TextField` 在表单、工具栏和对话框中的布局职责不同。

## 操作步骤

### 1. 一维关系优先 VStack/HStack

标题—字段—按钮—提示是纵向任务流，用 `VStack`；工具栏或成对按钮是横向协作，用 `HStack`。`spacing` 表示相邻项目节奏，对齐属性决定交叉方向位置。需要中间区域扩展时，为它设置伸缩，而不是给两侧大量固定空白。

### 2. 等宽二维单元使用 Grid

当前 `Grid(columns)` 把可用宽度分成指定数量的**等宽列**，适合计算器键盘和同权指标卡。它没有逐列宽度或权重参数。标签窄、输入区宽的表单应使用多行 `HStack`：标签固定宽度，输入控件 `.flex()`；不要在 API 中寻找不存在的列模板。

### 3. 不定数量标签使用 FlowRow

筛选标签、徽标等项目按可用宽度换行时用流式容器。它适合高度接近的小项目，不适合代替数据表或复杂卡片网格。

### 4. 同一区域前后叠放使用 ZStack

当两个或更多子组件确实要共用一块区域时，用 `ZStack`。常见例子是正文上方的状态角标、预览图上方的加载提示，或主内容上方的通知层。先声明的组件在下，后声明的组件在上；上层组件也会先收到点击事件。

不要把它和 `Grid`、`FlowRow` 混用：`Grid` 给每个项目分配互不重叠的格子，`FlowRow` 在宽度不足时把后续项目换到下一行，`ZStack` 则让所有子组件在同一个框架中对齐。它不会自动生成行列，也不会解决内容溢出。若想让项目并排或自动换行，看到控件互相遮住就说明选错了容器。

调整窗口时可以直接观察结果：底层内容和上层角标始终在同一块区域内，角标保持设定的对齐位置并盖在正文之上。若上层只是提示而不应操作，还要确认它没有挡住下层点击；需要遮住整个窗口并接管输入的对话框，应使用 `Modal` 等专用控件，而不是把普通按钮随意叠在根布局上。

### 5. 内容超出视口再加滚动

普通设置页用 `ScrollView` 包裹完整内容树；大量固定高行使用 `LazyColumn`；左右主从工作区使用 `SplitView`。若产品要求工具栏或提交按钮始终固定，应在页面根部再分出固定动作区，而不是只把几个字段塞进一个高度不明确的内层滚动区。

若目标仍是设置表单，把 `app.run` 中原来的根 `VStack` 整体替换为下面的滚动内容。`name`、`accepted` 和 `message` 都沿用基页状态；校验动作保持原样。完整根替换避免出现“滚动字段只有不确定的剩余高度，标题初始帧反而被裁掉”的布局：

```cangjie role=patch
ScrollView {
    VStack(spacing: 12.vp) {
        Label("创建账户").bold()
        TextField(name)
        Checkbox("我已阅读并同意条款", accepted)
        Button("提交", {=>
            if (name.value.trimAscii().isEmpty()) {
                message.value = "请输入姓名"
            } else if (!accepted.value) {
                message.value = "请先同意条款"
            } else {
                message.value = "欢迎，${name.value}"
            }
        }).role(ButtonRole.Primary)
        Label(message.value)
    }.padding(20.vp)
}
```

如果常用窗口足够宽，希望标签与字段按等宽二维关系对齐，可以只把基页标题、输入与复选三行换成下面的网格，提交回调继续读取原来的 `name` 与 `accepted`，无须增加新模型：

```cangjie role=variation
VStack(spacing: 12.vp) {
    Label("创建账户").bold()
    Grid(2) {
        Label("姓名")
        TextField(name)
        Label("条款")
        Checkbox("我已阅读并同意条款", accepted)
    }.spacing(10.vp)
}
```

### 6. 用窗口边界测试

至少测试最小可用尺寸、常用尺寸和更大尺寸。观察文字是否被裁、操作是否仍可达、剩余空间是否分给正确区域。固定 `vp` 应用于按钮高度或局部间距，不应用来锁死整个窗口布局。

## 确认结果

缩小窗口时，重要操作仍可通过滚动或合理换行到达；放大时，主内容区吸收空间而不是所有控件无意义拉伸。纵向或横向栈保持线性阅读顺序，网格保持行列对齐，流式项目会换行，层叠内容仍共享同一区域，滚动内容仍可到达。Tab 顺序与阅读顺序一致，分栏拖动只改变两个工作区比例。若关闭窗口缩放后布局才正确，说明代码仍依赖偶然尺寸。

## 常见错误

- 所有页面都套 `VStack`，再靠固定宽度模拟列对齐。
- 给大列表只加 `ScrollView`，以为节点数量会自动减少。
- 用 `ZStack` 模拟网格或换行，结果控件互相遮住，上层还可能截走下层点击。
- 用数组索引决定动态项目身份，排序后状态跟错行。
- 把工具栏放进长滚动内容，用户滚到底才能执行主要操作。
- 过度使用 `Spacer` 把相关标签和输入推得很远。
- 为解决拥挤统一缩小字号，而不是减少同屏问题或重组面板。
